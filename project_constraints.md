# 项目约束 (Project Constraints)

> 本文件定义了项目的技术边界、开发规范和架构限制，是项目决策的核心参考文档。

## 1. 技术架构约束

### 1.1 静态站点架构
- ✅ **允许**：纯HTML/CSS/JavaScript实现
- ✅ **允许**：GitHub Pages托管
- ✅ **允许**：GitHub Actions自动化部署
- ❌ **禁止**：引入后端服务（Node.js/Python/PHP等）
- ❌ **禁止**：使用数据库（MySQL/MongoDB/SQLite等）
- ❌ **禁止**：引入前端框架（Vue/React/Angular等）

### 1.2 单文件架构限制
- ⚠️ **警告**：文章数量超过30篇时应考虑数据文件分离
- ⚠️ **警告**：文章数量超过50篇时必须重构为多文件架构
- ⚠️ **当前状态**：约1200行代码集中在单个index.html文件中

### 1.3 依赖管理
- ✅ **允许**：使用CDN引入外部库（如Fuse.js搜索库）
- ❌ **禁止**：使用npm/yarn/pip等包管理器管理前端依赖
- ❌ **禁止**：引入构建工具（Webpack/Vite/Parcel等）

## 2. 代码风格约束

### 2.1 HTML规范
- 使用语义化标签（`<header>`, `<main>`, `<article>`, `<section>`, `<footer>`）
- 缩进使用2空格
- 属性顺序：class → id → data-* → src/href → title/alt
- 自闭合标签不加斜杠（`<br>` 而非 `<br />`）

### 2.2 CSS规范
- ✅ **必须**：使用CSS变量（`:root`）管理主题色彩
- ✅ **必须**：采用暗色主题配色（背景#0d1117、文字#e6edf3、强调#58a6ff）
- ⚠️ **建议**：采用BEM命名规范（Block__Element--Modifier）
- ⚠️ **建议**：选择器嵌套不超过3层

### 2.3 JavaScript规范
- ✅ **必须**：使用ES6+语法（const/let、箭头函数、模板字符串）
- ✅ **必须**：函数职责单一，每个函数只做一件事
- ⚠️ **建议**：使用JSDoc注释添加函数说明
- ⚠️ **建议**：变量命名使用camelCase，常量使用UPPER_SNAKE_CASE
- ❌ **禁止**：全局变量污染（使用IIFE或模块化封装）

## 3. Git工作流约束

### 3.1 分支策略
- ✅ **必须**：`main`分支不受保护，可直接推送
- ✅ **必须**：新功能创建feature分支（命名：`feature/描述`）
- ✅ **必须**：Bug修复创建fix分支（命名：`fix/描述`）
- ⚠️ **建议**：使用Pull Request合并代码

### 3.2 Commit Message规范（Conventional Commits）
```
<type>(<scope>): <subject>

Types:
- feat: 新功能
- fix: Bug修复
- docs: 文档变更
- style: 代码格式调整
- refactor: 重构
- perf: 性能优化
- test: 测试相关
- chore: 构建/工具变动

Examples:
- feat(article): 添加新文章
- fix(ui): 修复移动端样式问题
- docs(readme): 更新项目说明
```

### 3.3 部署约束
- ✅ **必须**：推送到main分支触发GitHub Actions自动部署
- ✅ **允许**：手动部署到GitHub Pages
- ❌ **禁止**：在生产环境进行手动修改

## 4. 内容管理约束

### 4.1 文章数据规范
- ✅ **必须**：每篇文章包含以下字段：
  - `id`：唯一标识（递增整数）
  - `title`：文章标题
  - `account`：公众号名称
  - `date`：发布日期（YYYY-MM-DD格式）
  - `url`：原文链接（必须是mp.weixin.qq.com）
  - `tags`：标签数组（从预定义标签中选择）
  - `summary`：摘要（建议50-100字）
  - `keyPoints`：核心要点数组（建议5-8个，每个30-60字）
  - `repos`：关联仓库数组（可选）
  - `evaluation`：个人评价（可选）

### 6.1 外部链接
- ✅ **必须**：所有外部链接使用`target="_blank"`

### 6.2 用户输入
- ⚠️ **建议**：搜索框输入进行基本过滤

- ❌ **禁止**：在代码中硬编码任何密钥、token或密码
- ❌ **禁止**：提交包含个人敏感信息的文件

## 10. 自动化文章收录工作流

### 10.1 概述
本项目采用**URL驱动自动化流程**收录公众号文章。当用户提供文章URL时，系统自动完成：
1. 抓取文章内容
2. 解析标题、公众号、日期等元信息
3. 生成摘要和核心要点
4. 更新 index.html 数据文件
5. 提交到本地 Git 仓库

### 10.2 自动化脚本
- **脚本位置**：`scripts/collect_article.py`
- **运行环境**：Python 3.11+
- **依赖库**：requests, beautifulsoup4

### 10.3 使用方式
```bash
python scripts/collect_article.py <文章URL>
```

### 10.4 自动化流程
```
用户输入URL
    ↓
脚本抓取文章内容
    ↓
解析HTML提取：标题、公众号、日期、正文
    ↓
AI分析生成：摘要(50-100字)、核心要点(5-8个)、标签推荐
    ↓
更新 index.html（追加新文章到 articles 数组）
    ↓
自动执行 git commit 到本地仓库
    ↓
推送提示（等待用户确认后推送到远程）
```

### 10.5 数据生成规则
- **标题**：从文章 `<h1>` 或 `<title>` 提取，最多80字符
- **公众号**：从文章元信息或正文提取
- **日期**：当天日期（YYYY-MM-DD格式）
- **标签**：基于内容关键词自动匹配预定义标签（最多4个）
- **摘要**：文章开头100字或AI生成
- **要点**：正文前5段的核心观点提炼

### 10.6 Commit规范
- **消息格式**：`feat(article): 收录新文章 - 《文章标题》`
- **提交范围**：仅 `wechat_articles/index.html`
- **分支策略**：直接在 `main` 分支提交

### 10.7 错误处理
- URL无法访问：提示用户手动提供文章内容
- 解析失败：跳过无法解析的字段，留空供用户补充
- Git提交失败：输出错误信息，保存更改但不提交
